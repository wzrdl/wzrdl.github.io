# 1. IntelliSys：LLM Forgetting / ICML under review

这是最容易被深挖的一段。

## Q3：你说“Chance-Constrained Optimization for LLM Forgetting”，先用通俗话讲一下你们到底在解决什么问题。

**答法：**
我们解决的是：**大模型在持续后训练或领域适配时，学到新知识的同时会破坏旧知识**。
传统方法通常把这个问题当成一个确定性的约束优化问题，比如限制参数不要偏得太远，或者限制某些重要方向不要变化太大。
但我觉得这里有个问题：遗忘本质上不是一个完全确定的现象，它和数据分布、任务冲突、优化路径都有关系，所以更适合被建模为一种**概率风险**。
我们的方法就是把“遗忘不能超过某个阈值”从硬性的确定性边界，变成一个机会约束：在高概率下控制遗忘风险。这样做的好处是更贴近真实训练过程，也更适合处理大模型持续后训练时的不确定性。

---

## Q4：为什么传统 Hessian-based 或 parameter regularization 方法不够好？

**答法：**
我理解有三个主要问题。
第一，**它们通常假设局部二阶近似足够可靠**，但在大模型持续训练里，损失面非常复杂，这种近似不一定稳定。
第二，**它们常常把“重要参数”视为静态的**，但在 continual post-training 里，参数重要性会随着新任务变化。
第三，**它们对风险的表达是确定性的**，比如“参数变化不能超过多少”，但实际遗忘更像一个概率事件，不同 batch、不同任务顺序下表现不同。
所以我们想把它从“硬边界约束”往“风险控制”这个方向推进。
如果面试官继续追问，我会补一句：我们不是说 Hessian 方法没用，而是说它在 LLM continual post-training 这个场景下，表达能力和稳定性有局限。

---

## Q5：你说“遗忘最高降低 2.96%”，这个指标怎么定义？为什么不是 20% 这种很高的数字？

**答法：**
这个数字看起来不大，但在大模型持续后训练里其实是很实在的提升。
因为这里不是从一个很弱的 baseline 提到一个很强的模型，而是在已经比较强的 LoRA 和全参数微调基线上，去**抑制旧知识退化**。
遗忘的度量一般不是只看一个任务，而是看模型在旧任务集合上的性能下降，比如 MMLU 或其他保留集上的下降幅度。我们报告的是相对于基线方法的最大遗忘减轻幅度。
我会强调两点：
一是，这类问题的改进通常不会像分类任务那样出现十几个点的暴涨；
二是，如果你的目标是“尽量不忘”，哪怕是几个点，在大模型上也非常有价值。

---

## Q6：你这个工作里最大的技术难点是什么？

**答法：**
最大的难点不是“把模型训起来”，而是**怎么把遗忘这个现象形式化成可优化、可验证的目标**。
因为如果定义得太粗，它只是一个经验观察；定义得太理想化，又很难在大模型上跑通。
我做这部分时比较关注三件事：
第一，风险定义要能反映“旧知识退化”；
第二，优化形式不能太重，否则 7B–14B 模型无法承受；
第三，实验设计要能把“单纯没学到新任务”和“真的发生遗忘”区分开。
所以难点其实在“理论表达”和“训练可行性”之间找平衡。

---

## Q7：如果我要你一句话说清你的核心贡献，不要讲团队，只讲你自己，你怎么说？

**答法：**
我在这个项目里的核心贡献，是**把 LLM 持续后训练中的遗忘问题从传统的确定性参数约束，重构成一个概率风险控制问题，并完成了对应的实验验证与效果分析**。

---

## Q8：你怎么证明这不是单纯因为你调参更好？

**答法：**
这是一个必须回答的问题。
我会从三个层面回答：
第一，和标准 LoRA、全参数微调做直接对比；
第二，做消融实验，拆开风险约束、正则项、训练设置，看提升到底来自哪里；
第三，看多个模型规模和多个评测集，而不是只在一个设置上赢。
如果一个方法只在单一 setting 里提升，很可能只是调参；但如果它在不同模型和任务上都有一致趋势，可信度会更高。
所以我会用“跨设置一致性 + 消融”来回答“不是调参偶然”的质疑。

# 2. BAGL：GraphRAG 与 LLM 微调医疗问答

这一段是你最强的“产业 + 医疗 + LLM 落地”经历之一。

## Q9：你为什么要做 GraphRAG，而不是普通 RAG？

**答法：**
因为医疗问答里很多信息不是孤立文档片段，而是**实体—关系—约束**。
普通 RAG 更擅长从文本中召回相关片段，但它不天然擅长表达“这个症状和哪个部位、哪类事件、哪条证据链相关”。
GraphRAG 的好处是，它能把知识结构显式表示出来，让模型回答时不是只依赖语义相似，而是也依赖结构约束。
在医疗这种高风险场景里，这很重要，因为我们更希望模型“有依据地回答”，而不是“语言上像对的”。
所以我们用了 **Neo4j Cypher 结构化查询 + FAISS 稠密检索** 的混合方式：图负责结构和约束，向量负责召回覆盖。

---

## Q10：你说 GraphRAG 显著缓解幻觉，具体是怎么缓解的？

**答法：**
本质上有两层。
第一层是 **检索前约束**。图谱查询能把候选知识限制在某些实体、关系和路径上，减少模型拿到无关上下文的概率。
第二层是 **回答时的事实支撑**。模型生成答案时，不是只依赖参数记忆，而是依赖结构化证据和检索证据。
这样做不能保证绝对不幻觉，但能显著减少“无依据编造”。
如果面试官继续问，我会再补一句：GraphRAG 更适合“需要多跳关系和显式可追踪证据”的问答场景。

---

## Q11：为什么还要微调 LLaMA-3 和 Mistral-7B？直接 RAG 不够吗？

**答法：**
单纯 RAG 可以补知识，但它不一定能让模型学会**如何使用这些领域知识来回答问题**。
在医疗场景里，我们发现光给上下文还不够，模型仍然可能：

* 不按专业风格回答
* 不遵守证据边界
* 不够稳定地引用领域概念
  所以我们做了自建垂直领域指令数据集，再用 QLoRA 微调，让模型在“回答形式、术语使用、基于证据作答”这些方面更贴合场景。
  我会把它理解为：RAG 提供外部知识，微调塑造使用知识的行为。

---

## Q12：事实一致性提升 20%，怎么评估的？

**答法：**
事实一致性不是看回答流不流畅，而是看回答是否和参考知识、检索证据、标注答案一致。
实际评估时，通常会设计内部 benchmark，包含领域问题、标准答案或专家参考，再从一致性、覆盖率、是否引入错误事实等维度去评。
我会强调两点：
第一，这是**内部评估基准**上的提升，不会把它说成公开 benchmark 的 SOTA；
第二，我们关注的不是“写得更像人”，而是“是否忠实于知识与证据”。

---

## Q13：GraphRAG 系统里最难的不是模型，而是什么？

**答法：**
最难的是**知识组织和查询路径设计**。
因为如果图谱 schema 设计得不好，Cypher 查询会很脆弱；如果检索粒度切分不好，向量检索会召回很多表面相关但实际上没用的内容。
所以真正难的是把知识图谱、向量库和问答行为连接起来，让它们在工程上可维护，在回答上可控。

# 3. Scientific Reports：临床知识图谱抽取

这段会被问你是否真的懂 NLP 信息抽取和统计建模。

## Q14：为什么要用 GPT-4o 做信息抽取，而不是传统 NER/RE 模型？

**答法：**
因为临床病历是高度非结构化、表达多样且上下文依赖强的文本。
传统 NER/RE 模型在定义明确、标注充足的场景下很强，但在这个任务里，标注成本高，概念边界复杂，而且我们需要的不只是实体识别，还包括较复杂的结构化映射。
所以我们选择用 GPT-4o 做受约束的信息抽取，同时通过严格 prompt 约束去减少格式漂移和抽取不一致。
简单说，LLM 给了我们更强的泛化抽取能力，而 prompt constraints 是为了把这种能力变成可控输出。

---

## Q15：既然 LLM 会幻觉，你怎么保证抽取结果可信？

**答法：**
我们不是“直接相信 LLM”，而是通过多层约束降低风险。
第一，用严格的 schema 和 prompt 约束输出格式。
第二，对实体和脑区做标准化映射，避免自由文本漂移。
第三，在图构建阶段加入统计平滑和后处理，而不是原样把所有抽取结果当真。
所以系统不是“LLM 一步到位”，而是“LLM 抽取 + 标准化 + 统计建模 + 图构建”的流水线。

---

## Q16：Poisson-Gamma 贝叶斯平滑为什么适合这个场景？

**答法：**
因为这里面对的是**长尾、稀疏的共现数据**。
如果直接用原始频次建图，低频共现可能完全不稳定，高频节点又会主导整个图。
Poisson-Gamma 平滑本质上是在计数数据上加入先验，得到更稳健的后验估计，这对稀疏共现关系特别有帮助。
我会这样解释：它的价值不是“让结果更漂亮”，而是**避免图谱被偶然出现的低频噪声关系带偏**。

---

## Q17：GMM 在这里起什么作用？

**答法：**
我们用 GMM 不是为了炫技，而是为了对**潜在关系强度做聚类**。
直观上，不同关系并不一定只是“强/弱”两档，它们可能天然形成几个分布簇。
GMM 的作用是帮助我们从平滑后的强度分布中识别更有意义的群组，从而筛出更高置信度的关系边。
它相当于在“统计估计”之后，再加一层“结构筛选”。

# 4. IISE 审稿中：KG-based RL

这段很容易被问你到底懂不懂 RL，而不是只会说 PPO。

## Q18：你为什么把癫痫发作进展建模成 MDP？

**答法：**
因为我们关注的是一个**序列决策/状态转移过程**，不是单点分类。
癫痫发作的进展本身带有时序演化特征，症状之间有转移关系，所以把它抽象成状态、动作、转移和奖励是自然的。
MDP 框架的好处是它允许我们显式建模“当前症状状态会如何影响下一步路径选择”，而不是只做静态预测。

---

## Q19：为什么用 PPO，不用 DQN 或 supervised learning？

**答法：**
如果状态和策略空间比较复杂，而且策略更新需要稳定，PPO 通常是一个更稳妥的选择。
DQN 更适合离散动作且 value-based 设定比较清晰的情况，但在我们的任务里，策略约束和训练稳定性更关键。
至于为什么不是直接监督学习，因为我们不仅关心某一步分类对不对，还关心**整个路径/传播过程是否合理**。
PPO 给了我们一个优化整段行为质量的框架。

---

## Q20：奖励塑形怎么做的？为什么不会把模型带偏？

**答法：**
奖励塑形的核心思路，是把知识图谱嵌入提供的结构信息注入到奖励中，让策略更倾向于走符合图谱语义和连通性约束的路径。
但奖励塑形确实有风险，最大的问题就是“过度引导”，把模型强行推向图谱偏好。
所以设计上要避免 reward 过强，不能让图谱先验完全盖过环境反馈。
我会告诉面试官：奖励塑形是为了帮助探索，而不是替代真实目标。

---

## Q21：你说预测回报提升 40%，这个提升来自 PPO 还是来自图谱奖励？

**答法：**
这正是需要通过消融回答的问题。
理论上提升可能来自三块：

* RL 框架本身比静态方法更适合序列传播建模
* PPO 比其他训练方式更稳定
* KG embedding 奖励塑形提供了有用先验
  所以如果我来回答，我会说：不能把 40% 全部归因于 PPO，必须通过去掉图谱奖励、替换训练策略、改变连通性约束等 ablation 才能拆清来源。
  这个回答会让面试官觉得你对归因问题是清醒的。

# 6. Miniagent

## Q24：你这个 Miniagent 和现有 agent framework 有什么区别？

**答法：**
我做 Miniagent 的目标不是做一个功能最全的 agent 框架，而是做一个**面向 RL 训练的最小可扩展骨架**。
很多现有框架更偏应用 orchestration，适合搭业务流程；而我更关注的是在 rollout 过程中，agent 如何调用工具、如何收集轨迹、如何接入 RL 算法比如 GRPO。
所以它的重点不在 UI，而在训练闭环。

---

## Q25：为什么是 GRPO？它相比 PPO 的优势是什么？

**答法：**
我选择 GRPO，主要是因为它更适合某些基于相对反馈或组内比较的训练设定，尤其在 LLM agent 任务里，有时候 reward 不是绝对标量，而更像是相对优劣。
不过我不会在面试里把它吹得过头。我会说：
PPO 是更通用、更成熟的强化学习优化框架；
GRPO 的价值在于，它可能更适合某些语言模型对比式优化设定。
如果对方继续问，我会把重点放在 rollout 设计、轨迹采样、工具调用与 credit assignment 上，而不是空谈算法名字。

## Q26：agent 调工具最大的难点是什么？

# 7. Mini-vLLM

这是非常适合系统岗深挖的一段。

## Q27：什么是 PagedAttention？为什么它重要？

**答法：**
PagedAttention 的核心思想是把 KV cache 从一整块连续内存，变成类似分页管理的块式结构。
这样做的重要性在于：LLM 推理时不同请求长度不同，如果要求连续分配，很容易出现显存碎片和内存浪费。
分页式管理可以让 KV cache 更灵活复用，也更适合动态 batch 和长上下文场景。
所以它不是单纯的 attention 优化，而是**推理系统内存管理设计**的一部分。

---

## Q28：你说 Block-based KV Cache 管理，具体解决了什么问题？

**答法：**
它主要解决两个问题。
第一，**显存碎片**。请求长度不一致时，连续分配会留下很多空洞。
第二，**长上下文下的扩展性**。当上下文越来越长时，KV cache 是主要内存开销，块式管理更容易做复用和调度。
我会把这个回答成一句话：
它提升的不是某一个 kernel 的速度，而是整个推理系统在复杂请求负载下的资源利用率。

---

## Q29：你自己实现 Mini-vLLM，最难的部分是什么？

**答法：**
最难的部分是**把“模型逻辑”与“系统调度逻辑”分开**。
模型前向本身不难理解，难的是如何组织 request、cache block、decode step 和 memory mapping，让它在动态请求场景里还能稳定工作。
这个项目让我最大的收获是：推理优化并不是单点算子优化，而是内存、调度、batching 和 attention 共同作用的系统问题。

# 8. Smarter Doctor agent

这段非常适合系统设计 + 落地问法。

## Q30：你这个实时语音医疗 agent 的端到端架构怎么讲？

**答法：**
我会按请求流来讲。
用户语音进入系统后，前端和后端通过 WebSocket 做流式传输，后端对输入做处理，并通过工具编排调用检索或数据服务，比如 Elasticsearch、BigQuery；大模型能力通过 Vertex AI 接入 Gemini；电话相关能力由 Twilio 支撑；整个服务用 FastAPI + Next.js 搭建，并通过 Docker 和 GitHub Actions 做 CI/CD。
这样讲能同时体现：实时交互、工具编排、云上部署和可持续交付。

---

## Q31：为什么要用 WebSocket，不用普通 HTTP？

**答法：**
因为语音交互是强实时、双向连续的数据流场景。
HTTP 更适合一次请求一次响应，而 WebSocket 更适合持续推流、低延迟交互和服务端主动返回中间结果。
在语音 agent 里，如果你用普通 request-response 模式，用户体验会差很多。

---

## Q32：医疗场景为什么难？

**答法：**
医疗场景难在三点。
第一，容错率低，答错的代价高。
第二，信息来源复杂，需要结合结构化和非结构化数据。
第三，用户期待的不只是“像医生说话”，而是“有依据、可追踪、尽量少幻觉”。
所以我做这个项目时会特别强调工具增强、检索支撑和系统可控性，而不是只追求对话自然。

# 9. Kernel K-Means GPU Accelerator

## Q33：你为什么能把 Kernel K-Means 写成稀疏线性代数形式？

**答法：**
因为 Kernel K-Means 本质上很多计算都可以重写成矩阵操作，而一旦数据或中间结构存在稀疏性，就能借助稀疏线性代数库提高效率。
把算法重写成适合 GPU 的矩阵/稀疏操作后，就可以更好利用 cuSPARSE 和 cuBLAS，而不是停留在逐点计算视角。

---

## Q34：全局内存合并访问和 shared memory tiling 具体为什么有效？

**答法：**
因为 GPU 性能很大程度受内存访问模式影响。
全局内存合并访问可以减少访存事务，提高带宽利用率；
shared memory tiling 则是把重复访问的数据放进更快的片上存储，减少全局内存访问次数。
所以这两个优化本质上都是在解决“算得快之前先喂得饱”的问题。

---

# 10. 医学提示词自动优化

## Q35：什么叫“带动量的文本梯度式更新”？

**答法：**
它不是传统意义上的可微梯度，而是把 prompt 当作可迭代优化对象，通过模型反馈、验证结果和启发式更新方向去逐步改进。
“带动量”表示我们不是每一步都完全被当前反馈牵着走，而是保留一部分历史更新方向，让 prompt 优化更稳定。
这个说法很容易被追问，所以你回答时一定要强调：这是**类梯度/文本梯度式更新**，不是对 prompt token 真做反向传播。

---

## Q36：为什么还要做贝叶斯验证？

**答法：**
因为 prompt 优化很容易过拟合某些样例。
贝叶斯验证的价值在于，不只看某次改动在当前样本上是否变好，而是更稳健地估计“这个改动是否大概率真的有效”。
换句话说，它帮助我们避免把偶然有效的 prompt 更新误判成稳定进步。
